# Creational Patterns

**1. Singleton**

- _Проблема:_ Полезно для доступа к какому-то общему ресурсу, например, БД.
- _Решение:_ Создать единственный объект класса и хранить его в статической переменной. Запретить создание второго объекта этого класса.
- _Реализация:_ Скрыть конструктор по умолчанию и создать публичный статический метод, который и будет контролировать жизненный цикл объекта-одиночки.
- _Плюсы:_ уменьшение кол-ва использования памяти приложением.

**2. Prototype**

- _Проблема:_ Не каждый объект удастся скопировать таким образом, ведь часть его состояния может быть приватной, а
  значит — недоступной для остального кода программы.
- _Решение:_ Паттерн поручает создание копий самим копируемым объектам.
- _Реализация:_ Вводим общий интерфейс для всех объектов, поддерживающих клонирование.

**3. Factory method**

- _Проблема:_ . Код зависит от конкретных классов. Если ты создаёшь объекты через new, то чтобы поменять реализацию, придётся менять код везде — есть new Circle() и new
  Square(), но завтра надо будет создавать Triangle. Придётся менять много мест.
- _Решение:_ Создать общий интерфейс для объектов и использовать фабричный метод, который сам решает, какой конкретный класс создать.
- _Реализация:_ Создаём абстрактный класс или интерфейс, в котором есть метод create(), а конкретные классы переопределяют этот метод и возвращают нужный тип (Circle, Square).

**4. Abstract factory**

- _Проблема:_ Нужно создавать набор взаимосвязанных объектов, но ты не хочешь жёстко привязываться к их конкретным
  реализациям. В зависимости от операционной системы тебе нужно создавать: Checkbox и Button. И все они должны быть в одном стиле (MacOS или Windows).
- _Решение:_ Создать одну "абстрактную фабрику", которая возвращает целый набор объектов, а конкретные реализации этой фабрики возвращают нужные типы.
- _Реализация:_
    - пишем абстрактнеы Checkbox и Button. От которых будут унаследованы специфичные для каждой ОС типы.
    - Пишем абстрактную фабрику, реализациями которой будут фабрики, специфичные для каждой ОС.
      Плюсы:
    - Централизованное управление семейством продуктов.
    - Упрощает замену семейств продуктов; гарантирует сочетаемость продуктов.

**5. Builder**
- _Проблема:_ Создание сложных объектов с большим количеством параметров может стать нечитабельным и неудобным.
- _Решение:_ Вынести процесс создания объекта в отдельный объект-построитель, который по шагам собирает нужные части.
